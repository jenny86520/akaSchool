接下來我來一一介紹：

.git/

為隱藏的資料夾，需開啟檔案總管的「顯示隱藏的項目」才看的到

存放 git 版控的相關資料，大致上可以不用理會它，畢竟版控不是目前我們要學的:P

只要記得不要刪掉它！不然就要重新連到版控上了（即遠端分支

如果你要再練習一次如何連上版控，那刪吧，加油((x

dist/

部屬後新建的資料夾，存放部屬後可發行的檔案

執行「ng build」或「ng build --prod」指令後，所有檔案將壓縮成瀏覽器看得懂的語言（html、javascript、css）

因此只要將裡面的檔案都丟到網站伺服器（我們使用 GitHub Pages，所以就是丟上 GitHub Pages），瀏覽器讀取時就能運作囉！

e2e/

用來存放 E2E（End-to-End）測試程式碼

單元測試與 E2E 測試比較圈外(?)，有興趣可以參考：Angular單元測試與E2E測試

node_modules/

為所有套件的存放地

通常檔案大小很大，版控時會 ignore 的資料夾（寫在 .gitignore 中）

src/

主要會更動的檔案都會放在 src 資料夾下

底下簡單介紹一下 src/ 底下初始的檔案：



app/

存放所有撰寫的程式碼（Component、Module、Pipe、Service...）

底下簡單介紹一下 src/app/ 底下初始的檔案：

app.component.css / app.component.html / app.component.ts / app.component.spec.ts

Angular CLI 預設建立的 root component

app.component.spec.ts 為測試用檔案，基本上可以不用理它

app.module.ts

預設的 root module，用來告訴 Angular 有哪些 Components、Modules、Services...

讓 Angular 知道如何 assemble the application

app-routing.module.ts

設定路由（Router）的地方

新建 Angular 專案時，如果有選擇新建 routing，將會自動建立的檔案（上一篇教學有選擇 yes 唷！）


assets/

放圖片或者其他需要用的材料

底下有一個初始檔案「.gitkeep」，為版控用檔案

上傳到版控時，空的資料夾預設是不會進入版控的，而資料夾中添加「.gitkeep」檔後，即使資料夾為空亦會進版控

environments

設定 Angular 程式碼會用到的參數，類似 Web.config

執行或部屬時，預設使用 environment.ts 檔中的參數，要產生不同的 environment 的話，參考命名規則為 environment.xxx.ts

執行或部屬時加上 xxx 的參數，就可指定到該 environmnet（ng serve --xxx / ng build --xxx）

底下初始會有兩個檔案（environmnet.ts、environmnet.prod.ts），上一篇我們教的部屬指令，即是使用 environmnet.prod.ts 檔的參數

（environments 檔案的詳細介紹：0x06 ）

main.ts

這裡只要知道它「bootstrapModule(AppModule)」的部分，代表它指名從「AppModule」開始唷～

polyfills.ts

輔助瀏覽器的推薦套件清單，非常饒舌((x

顧名思義，就是當需要支援比較老舊的瀏覽器時（IE9.10.11），需要加載檔案中的套件，才能確保功能皆能正常運作

方法很簡單，打開檔案你會發現滿滿的註解，慢慢看用心看，就知道在什麼時候需要開啟哪個套件了

記得功能無法運作時，可以到此檔案看看是不是瀏覽器給你搞鬼喔！

styles.css

全域的裝飾樣式（CSS），存放整個專案都可以共享的 CSS 樣式，Over

test.ts

單元測試（Unit Tests）的大門，存放單元測試的設定和起始點，Over


.editorconfig

設定專案中檔案的風格，像是縮排、tab 的寬度以及 EOL 字元等

幫助開發者使用不同的 IDEs 時，保持檔案格式一致性的設定檔

.gitignore

設定上傳版控時要忽略的檔案，列於其中的檔案，將不會進入版控，Over

angular.json

Angular CLI 的設定檔（官方詳解：Angular CLI Config Schema）

browserslist

為了使編譯器能夠編出不同瀏覽器皆看得懂的設定

即給一些工具（Autoprefixer、Stylelint、babel-preset-env 等）引來使用，以避免花太多時間在為不同工具調整成同樣的目標瀏覽器配置

聽不懂嘛？那就看看這篇介紹，詳盡又淺顯易懂：什麼是 Browserlist？設定 Autoprefixer、Stylelint 來支援你想要的任何瀏覽器！

karma.conf.js

Karma test 的單元測試（Unit Tests）設定，暫先不討論測試～

package.json

Angular 的所有指令&版本設定檔（指令、套件版本、測試版本）

上一篇有提到「ng server」與「npm start」的差異，前者使用全域 Angular CLI 版本編譯，後者使用專案中 Angular CLI 版本

在此檔中，能找到「"@angular/cli": "~8.3.1"」設定，即專案版本（為專案建立時使用的版本，除非有再更新才會更改）

區分全域跟專案的 CLI 版本，用意是即使很舊的專案也能夠不失真的編譯運行

以及沒有安裝全域 CLI，也能使用專案中的 CLI 編譯。像是電腦並無安裝 Angular CLI，而從網上或是他人手中獲得的 Angular 專案，能直接執行「npm start」編譯運行

非常方便呢！

package-lock.json

紀錄第一次建立好專案時的所有版本設定檔（套件版本、測試版本）

在更新時（像是從 1.0 跳到 2.0），可能會改變一些套件的版本

此檔算是做歷史紀錄，當更新換版本出錯誤時，還能找回原始版本

詳細介紹請看這：package-lock.json 有什麼用，淺談 Yarn, NPM5 與 npm shrinkwrap

README.md

版控的說明文件，用來寫一些說明資料，於 GitHub 上專案首頁會看見的文件

.md 檔的撰寫的官方規格：Mastering Markdown

tsconfig.app.json / tsconfig.json / tsconfig.spec.json / tslint.json

TypeScript 編譯器設定檔，詳細解說可以參考：Angular 2筆記：tsconfig.json 設定，有興趣再研究吧～

tslint.json

Codelyzer（用以保持 code style 的一致性）和 TSLint 的設定檔，有興趣再研究吧～
